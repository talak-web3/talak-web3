This guide covers Sign-In with Ethereum (SIWE) authentication. For Next.js App Router, use the dedicated [Next.js integration](/docs/nextjs) (`toNextJsHandler`, `nextCookies`, `getSession`).

## Overview

The SIWE flow:

1. Client requests a nonce from the server
2. User signs an EIP-4361 message with their wallet
3. Server verifies the signature and creates a session

## Auth API routes

When you mount `app.handler` at `/api/auth/*` (the default), these routes are available:

| Route               | Method    | Description                                    |
| ------------------- | --------- | ---------------------------------------------- |
| `/api/auth/nonce`   | POST      | Generate SIWE nonce                            |
| `/api/auth/login`   | POST      | Sign in; sets httpOnly cookies when configured |
| `/api/auth/logout`  | POST      | Revoke session                                 |
| `/api/auth/refresh` | POST      | Rotate tokens                                  |
| `/api/auth/verify`  | GET, POST | Verify access token                            |
| `/api/auth/session` | GET       | Get session                                    |
| `/api/auth/jwks`    | GET       | JSON Web Key Set                               |

## Server setup

```ts title="talak.config.ts"
import { talakWeb3 } from "talak-web3";
import {
  InMemoryNonceStore,
  InMemoryRefreshStore,
  InMemoryRevocationStore,
} from "@talak-web3/auth";

export const app = talakWeb3({
  chains: [
    /* ... */
  ],
  auth: {
    nonceStore: new InMemoryNonceStore(),
    refreshStore: new InMemoryRefreshStore(),
    revocationStore: new InMemoryRevocationStore(),
    accessTtlSeconds: 900,
    refreshTtlSeconds: 604_800,
  },
});
```

Use Redis-backed stores in production — see [Installation](/docs/installation#production-redis-store).

## Sign-in flow

### 1. Get nonce

```ts
const response = await fetch("/api/auth/nonce", {
  method: "POST",
  credentials: "include",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ address }),
});
const { nonce } = await response.json();
```

### 2. Sign message

Construct and sign an EIP-4361 SIWE message with the server-provided nonce.

### 3. Login

```ts
await fetch("/api/auth/login", {
  method: "POST",
  credentials: "include",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ message, signature }),
});
```

Or use `TalakWeb3Client` with `baseUrl: "/api"` when the handler is at `/api/auth`.

## Verify session

### Next.js Server Components

```tsx title="app/dashboard/page.tsx"
import { getSession } from "talak-web3/nextjs";
import { app } from "@/talak.config";

const session = await getSession(app);
if (!session.isAuthenticated) redirect("/login");
```

### API routes / other frameworks

```ts
const response = await fetch("/api/auth/session", {
  credentials: "include",
});
const session = await response.json();
```

## Sign out

```ts
await fetch("/api/auth/logout", {
  method: "POST",
  credentials: "include",
});
```

## Token lifecycle

### Access token (JWT)

- **Format**: Signed JWT
- **Expiry**: 15 minutes (default)
- **Usage**: `Authorization: Bearer <token>` or `talak_web3_access` httpOnly cookie

### Refresh token

- **Format**: Opaque string (hash stored in Redis)
- **Expiry**: 7 days (default)
- **Usage**: `POST /api/auth/refresh` or `talak_web3_refresh` httpOnly cookie

### Token rotation

Refresh tokens are one-time use: the old token is revoked before new tokens are issued.

## Security guarantees

### Replay prevention

- Nonces are atomic (single-use)
- Redis-backed stores enforce atomic GET+DEL across instances

### CSRF protection

Double-submit cookie pattern on mutating auth requests when CSRF cookies are enabled.
